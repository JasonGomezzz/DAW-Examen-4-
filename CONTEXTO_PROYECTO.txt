DETALLE TÉCNICO DE LO CONSTRUIDO
=====================================================================

--- ms-usuarios (puerto 8081) ---
Entidad Usuario: id, nombre, email (unique), password (BCrypt), rol (enum CANDIDATO, ADMIN)
Endpoints:
  POST /api/usuarios/registro   -> público, crea usuario, hashea password con BCrypt
  POST /api/usuarios/login      -> público, valida credenciales, devuelve JWT
  GET  /api/usuarios/{id}       -> requiere JWT válido (cualquier rol)
  GET  /api/usuarios            -> requiere JWT con rol ADMIN
Es el ÚNICO servicio que EMITE JWT (JwtUtil.generarToken). Los otros dos solo VALIDAN.
Paquetes: controller, service/service.impl, repository, entity, dto, security, exception

--- ms-vacantes (puerto 8082) ---
Entidad Vacante: id, titulo, descripcion, requisitos, salario, estado (ABIERTA/CERRADA),
fechaPublicacion
Endpoints:
  POST   /api/vacantes            -> solo ADMIN
  PUT    /api/vacantes/{id}       -> solo ADMIN
  DELETE /api/vacantes/{id}       -> solo ADMIN
  GET    /api/vacantes/{id}       -> requiere JWT (cualquier rol) — usado también por Feign
                                      desde ms-postulaciones
  GET    /api/vacantes            -> requiere JWT (cualquier rol)
  GET    /api/vacantes/abiertas   -> PÚBLICO, sin JWT
Solo VALIDA JWT (no emite), usa el mismo secreto compartido que ms-usuarios.

--- ms-postulaciones (puerto 8083) ---
Entidad Postulacion: id, idVacante, idUsuario, estado (PENDIENTE/EN_REVISION/ACEPTADO/RECHAZADO),
fechaPostulacion
Endpoints:
  POST  /api/postulaciones                    -> requiere JWT, valida idVacante e idUsuario
                                                   vía Feign contra ms-vacantes y ms-usuarios
  PATCH /api/postulaciones/{id}/estado        -> solo ADMIN
  GET   /api/postulaciones/usuario/{idUsuario} -> requiere JWT (historial)
Paquetes adicionales frente a los otros 2:
  client/   -> UsuarioClient y VacanteClient (interfaces @FeignClient)
  config/   -> FeignClientConfig: RequestInterceptor que reenvía el header Authorization
               de la petición entrante hacia las llamadas Feign salientes (así ms-vacantes/
               ms-usuarios también reciben el JWT del usuario original)
Resilience4j (io.github.resilience4j:resilience4j-spring-boot3 v2.4.0):
  - ValidacionServiceImpl.validarVacante() / validarUsuario() están anotados con
    @Retry + @CircuitBreaker (nombres de instancia: "vacanteService", "usuarioService")
  - Si el servicio dependiente está caído: reintenta (3 intentos), si sigue fallando y se
    supera el umbral, el circuito se ABRE -> responde 503 inmediatamente con el mensaje
    "No fue posible consultar el servicio. Intente nuevamente." (viene de
    config-repo/application.yml -> app.mensaje-error-servicio, NO hardcodeado)
  - Un 404 real (feign.FeignException$NotFound, o sea "la vacante/usuario no existe") está
    EXCLUIDO del conteo de fallos del circuit breaker y del retry (ignoreExceptions), y se
    traduce a un 404 normal (ResourceNotFoundException), no al mensaje de fallback. Esto
    evita confundir "no existe" con "servicio caído".
  - Config completa de circuit breaker / retry vive en config-repo/ms-postulaciones.yml


--- SEGURIDAD JWT (cómo funciona entre los 3 servicios) ---
- jjwt 0.12.6 (jjwt-api, jjwt-impl, jjwt-jackson)
- HS256, secreto compartido puesto IGUAL en los 3 archivos:
    config-repo/ms-usuarios.yml
    config-repo/ms-vacantes.yml
    config-repo/ms-postulaciones.yml
  (clave: jwt.secret, jwt.expiration: 86400000 ms = 24h)
  NO se tocó config-repo/application.yml (archivo prohibido) — por eso el secreto está
  duplicado en los 3 archivos en vez de estar centralizado ahí.
- Claims del token: subject = email, claim "id" = id de usuario, claim "rol" = CANDIDATO|ADMIN
- Cada servicio tiene su propio JwtAuthenticationFilter (OncePerRequestFilter) que lee el
  header "Authorization: Bearer <token>", valida firma, y mete un
  UsernamePasswordAuthenticationToken en el SecurityContext con authority "ROLE_<rol>"
- SecurityConfig de cada servicio define qué rutas son públicas / requieren rol ADMIN /
  requieren solo estar autenticado


--- BASE DE DATOS ---
Cada microservicio tiene su propia BD MySQL (nada compartido):
  db_usuarios, db_vacantes, db_postulaciones
Credenciales: root / root, localhost:3306, createDatabaseIfNotExist=true (Hibernate las
crea solas, ddl-auto: update). database.sql en la raíz es un script de respaldo con el
mismo schema + datos de prueba, por si se quiere poblar una BD limpia a mano.

PROBLEMA TÉCNICO QUE APARECIÓ Y CÓMO SE RESOLVIÓ
=====================================================================
Al compilar ms-usuarios la primera vez, TODO lo que dependía de Lombok (@Data, @Builder,
@Slf4j, etc.) fallaba con "cannot find symbol" para getters/setters/builders/log que
nunca se generaban.

Causa raíz: desde JDK 23 en adelante, javac deshabilita por defecto el descubrimiento
automático de annotation processors en el classpath (antes bastaba con tener lombok como
dependencia). El JDK usado en la máquina al momento de compilar era JDK 24.

Fix aplicado en los 3 poms nuevos (ms-usuarios, ms-vacantes, ms-postulaciones):
  <plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-compiler-plugin</artifactId>
    <configuration>
      <annotationProcessorPaths>
        <path>
          <groupId>org.projectlombok</groupId>
          <artifactId>lombok</artifactId>
          <version>1.18.38</version>
        </path>
      </annotationProcessorPaths>
    </configuration>
  </plugin>

Esto fuerza a que se habilite el procesamiento de anotaciones explícitamente, así que
funciona tanto en JDK 21 como en JDK 23+. Con Java 21 ya configurado (como dijiste que
tienes ahora) probablemente ni hacía falta el workaround, pero no molesta dejarlo — es
inofensivo en cualquier JDK y hace el build más explícito/reproducible.

NOTA para mañana: si al compilar con tu JDK 21 real ves algún comportamiento raro
relacionado con Lombok, dime y lo revisamos, pero no debería pasar nada porque el fix
es compatible hacia atrás.

SESIÓN 2026-07-07 (Mac + OrbStack) — bugs reales encontrados al levantar en Docker
=====================================================================
Contexto: Jason clonó el repo (ya en GitHub, 1 commit) a su laptop Mac. Ahí usa OrbStack
(no Docker Desktop) como daemon Docker — 100% compatible con la CLI, sin cambios de
Dockerfile/compose por eso. `mvn` no está instalado en esa Mac pero no hace falta: los
Dockerfiles son multi-stage y compilan dentro del contenedor.

Al correr `docker compose up --build` aparecieron 5 bugs reales (nunca se había probado
en ejecución, ni en Docker ni en IntelliJ). Todos arreglados, quedaron los 6 servicios
+ MySQL healthy y los 5 clientes (api-gateway, ms-usuarios, ms-vacantes, ms-postulaciones,
config-server) registrados y UP en Eureka:

1. Puerto 8080 ocupado por otro proyecto Docker en la misma Mac (curso distinto,
   contenedor "springboot" del proyecto 2026-1-4c24-pi-2b). Fix: api-gateway ahora se
   expone en el host como 8090 (`"8090:8080"` en docker-compose.yml), puerto interno
   sigue en 8080. Actualizado en README, ATS-Postman-Collection.json (variable
   gatewayUrl) y este archivo. Si corres el Gateway suelto en IntelliJ sigue siendo 8080.

2. `spring.config.import: "configserver:http://localhost:8888"` hardcodeado en el
   application.yml empaquetado de api-gateway, ms-usuarios, ms-vacantes y ms-postulaciones.
   Al correr en Docker, esto intenta conectar a "localhost:8888" DENTRO del propio
   contenedor (nada escucha ahí) en vez de usar la variable de entorno SPRING_CONFIG_IMPORT
   ya seteada a config-server:8888 — un gotcha conocido de Spring Boot donde el config.import
   del classpath no es superado limpiamente por el de entorno. Fix: se cambió a
   `import: "configserver:"` (sin host) en los 4 application.yml, y se cambió la variable de
   entorno en docker-compose.yml de SPRING_CONFIG_IMPORT a SPRING_CLOUD_CONFIG_URI (esa sí
   es una property normal, con precedencia normal). Default sigue siendo localhost:8888 para
   quien corra en IntelliJ sin Docker.

3. config-server, corriendo en Docker, no encontraba `config-repo/` (esa ruta solo existe en
   el host, el Dockerfile de config-server YA tenía el comentario "en Docker se monta como
   volumen" pero el volumen nunca se agregó a docker-compose.yml). Sin esto, config-server
   respondía OK pero con propiedades vacías para todos los clientes (ningún error visible,
   simplemente no había nada que servir). Fix: se agregó
   `volumes: - ./config-repo:/config-repo:ro` al servicio config-server. No se tocó código
   de config-server ni el contenido de config-repo/application.yml ni api-gateway.yml.

4. MySQL 8 + `useSSL=false` sin permitir la llave pública → error real de conexión
   "Public Key Retrieval is not allowed" (el log "root@localhost created with an empty
   password" que preocupaba a Jason era en cambio un log de bootstrap inofensivo y conocido
   de la imagen mysql:8.0, no la causa real). Fix: se agregó `allowPublicKeyRetrieval=true`
   a la datasource URL en docker-compose.yml (los 3 microservicios) Y en
   config-repo/ms-usuarios.yml, ms-vacantes.yml, ms-postulaciones.yml (para que el mismo
   fix aplique también corriendo en IntelliJ contra MySQL local).

5. ms-postulaciones no arrancaba: `NoClassDefFoundError:
   io/github/resilience4j/spring6/fallback/RxJava3FallbackDecorator`. Causa raíz real:
   spring-cloud-dependencies (BOM, 2025.0.0) pisaba la versión transitiva de
   resilience4j-spring6 a 2.2.0 (esa versión no tiene la clase RxJava3FallbackDecorator),
   mientras resilience4j-spring-boot3 se declaraba en 2.4.0 — mismatch de versiones entre
   el autoconfig (compilado contra 2.4.0) y el jar real presente (2.2.0). Fix: se agregó
   el import de `io.github.resilience4j:resilience4j-bom:${resilience4j.version}` en el
   dependencyManagement de ms-postulaciones/pom.xml, ANTES del import de
   spring-cloud-dependencies (el orden importa: el primer import declarado gana si hay
   conflicto). Verificado con `mvn dependency:tree` que todo resilience4j-* quedó en 2.4.0.

6. api-gateway no se registraba en Eureka (seguía intentando `http://localhost:8761/eureka/`
   pese a tener EUREKA_CLIENT_SERVICEURL_DEFAULTZONE=http://eureka-server:8761/eureka/
   correctamente seteado). Causa: config-repo/application.yml (compartido, PROHIBIDO tocar)
   trae `eureka.client.service-url.defaultZone: http://localhost:8761/eureka/` para el modo
   IntelliJ/local, y ese valor remoto le ganaba a la variable de entorno para api-gateway
   específicamente (spring-cloud-config permite por defecto que la config remota
   sobreescriba variables locales: `spring.cloud.config.override-system-properties=true`
   por defecto). Nota: ms-usuarios/vacantes/postulaciones no sufren esto porque sus
   config-repo/ms-*.yml no declaran eureka.client, así que ahí no hay conflicto. Fix: se
   agregó `SPRING_CLOUD_CONFIG_OVERRIDE_NONE: "true"` como variable de entorno de
   api-gateway en docker-compose.yml (fuerza que lo local — env vars, classpath — SIEMPRE
   gane sobre lo remoto). No se tocó config-repo/application.yml ni api-gateway.yml.

Verificado con smoke test real vía curl a través del Gateway (localhost:8090):
